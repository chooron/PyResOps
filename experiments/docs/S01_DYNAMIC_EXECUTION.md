# S01 动态执行流程说明

本文档描述的是当前代码里的实际执行方式，对应实现文件主要是 `experiments/dynamic_experiment.py`、`experiments/evaluation_metrics.py` 和 `experiments/paper_experiment_runner.py`。

## 场景定位

`S01` 表示台汛期预泄场景。

当前动态实验里，这个场景的初始状态是：

- 当前水位 `157.5 m`
- 初始库容 `33.10 亿m3`
- 当前入流 `400 m3/s`
- 预报入流 `400 m3/s`
- 时步 `3 h`
- 总时长 `48 h`

## 这个场景里“指令更新”的核心方式

这个场景不是把四条指令一次性都发给 LLM，也不是每一轮都回到初始状态重算。

当前代码采用的是下面这套流程：

1. 先进入 `T0`，用初始状态请求一次 LLM。
2. LLM 只给出一个当前阶段的单值出库流量 `llm_outflow`。
3. 系统用这个常数出库流量，把状态推进到下一次触发时刻。
4. 进入 `T1` 时，LLM 看到的是推进后的真实状态，不是初始状态。
5. 同时，传给 LLM 的文本里只拼接当前这一阶段的新指令，不会把所有历史指令不断累加成长文本。

也就是说，`S01` 的动态性来自两件事：

- 状态是阶段间连续演变的。
- 指令是阶段性替换的。

## 每个阶段 LLM 实际看到什么

每进入一个阶段，系统都会做一次 `_apply_trigger()`。

这一步会把运行时状态写回场景字典：

- `current_level` 更新为上一阶段末水位
- `initial_storage` 更新为上一阶段末库容
- `initial_inflow` 更新为上一阶段末态里记录的当前入流
- `description` 改写成“原场景描述 + 当前阶段指令”
- `dynamic_stage` 写成当前阶段，例如 `T1`
- `dynamic_trigger` 写成当前指令文本

随后 `AgnoMCPExperiment.run_scenario()` 会基于这份更新后的场景调用工具链，因此工具读到的状态也是新状态，而不是静态 `SCENARIOS` 里的老状态。

## S01 的时间轴

### T0

- 触发时刻：`0 h`
- LLM 看到的状态：初始状态 `157.5 m`
- 指令文本：尽快把库水位从 `157.5 m` 降到 `156.5 m`
- 本阶段出库：由 LLM 返回一个 `llm_outflow`
- 推进窗口：`6 h`
- 达标判断：推进 `6 h` 后，`state_after_sim["level"]` 是否到达 `156.5 +/- 0.3 m`

这意味着 `T0` 不是只看回答文本，而是要先用该阶段出库流量把状态推演 6 小时，再判断是否达到目标水位。

### T1

- 触发时刻：`6 h`
- LLM 看到的状态：`T0` 推进 6 小时后的末状态
- 指令文本：加快预泄，接下来 6 小时内必须降到 `156.5 m`
- 推进窗口：`6 h`
- 达标判断：推进后水位是否到达 `156.5 +/- 0.3 m`

这里的关键点是：`T1` 不再从 `157.5 m` 开始，而是从 `T0` 的末态继续。

### T2

- 触发时刻：`12 h`
- LLM 看到的状态：`T1` 结束后的状态
- 指令文本：进一步降到 `155.5 m`，腾出额外库容
- 推进窗口：`12 h`
- 达标判断：推进后水位是否到达 `155.5 +/- 0.3 m`

这一步表示任务目标升级，系统不再只要求回到 `156.5 m`，而是要求继续降深。

### T3

- 触发时刻：`24 h`
- LLM 看到的状态：`T2` 推进后的状态
- 指令文本：台风减弱，恢复 `156.5 m` 目标并减小出库
- 推进窗口：剩余 `24 h`
- 达标判断：当前阶段的 `llm_outflow` 是否比上一阶段更小

这里要特别注意，当前代码里 `T3` 的 `pass_condition` 是 `direction=decrease`。

因此，`T3` 实际判定的是：

- 这次建议出库是否比 `T2` 小

而不是：

- 推进后水位是否严格回到 `156.5 m`

也就是说，`T3` 的“恢复 156.5 m 目标”目前主要体现在提示词语义里，自动化达标逻辑本身只检查“出库变小”这个方向性条件。

## 这个场景里状态如何向后传递

每个阶段结束后，系统调用 `_advance_state()` 做纯水量平衡推进。

推进逻辑是：

1. 用当前阶段的 `llm_outflow` 作为常数出库。
2. 用场景里的 `inflow` 作为常数入流。
3. 按 `3 h` 时步逐步更新库容。
4. 再通过库容曲线把库容换算回水位。

因此，`S01` 的下一阶段初始状态并不是 LLM 口头说了什么，而是上一个阶段在水量平衡下真实推演出来的末状态。

## 当前实现里的一个重要细节

配置里每个触发器都保留了 `sim_hours_before` 字段，但当前主流程真正用来确定阶段推进时长的，是相邻两次 `trigger_hours` 的差值。

对 `S01` 来说，实际推进区间是：

- `T0: 0 -> 6 h`
- `T1: 6 -> 12 h`
- `T2: 12 -> 24 h`
- `T3: 24 -> 48 h`

所以这个场景最终形成的是一条完整的动态时间轴，而不是四次彼此独立的单次实验。

## 最终输出里会记录什么

`S01` 运行完成后，结果里的 `stages` 会按 `T0~T3` 保存每一步的关键信息：

- 本阶段指令
- 阶段开始前状态 `state_before`
- LLM 返回的出库 `llm_outflow`
- 达标结果 `compliance`
- 推进后的状态 `state_after_sim`

因此，如果要检查 `S01` 的动态调整是否真实发生，最直接的看法就是对比：

- `stages[0]["state_before"]`
- `stages[1]["state_before"]`
- `stages[2]["state_before"]`
- `stages[3]["state_before"]`

这些水位不会相同，说明状态确实在阶段间演变。
