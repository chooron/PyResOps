# S05 动态执行流程说明

本文档描述的是当前代码里的实际执行方式，对应实现文件主要是 `experiments/dynamic_experiment.py`、`experiments/evaluation_metrics.py` 和 `experiments/paper_experiment_runner.py`。

## 场景定位

`S05` 表示梅台过渡期降水位场景。

当前动态实验里，这个场景的初始状态是：

- 当前水位 `160.0 m`
- 初始库容 `35.20 亿m3`
- 当前入流 `200 m3/s`
- 预报入流 `200 m3/s`
- 时步 `3 h`
- 总时长 `360 h`，也就是 `15 天`

## 指令更新方式

`S05` 的动态执行方式仍然是阶段式的：

1. 每个阶段只传当前状态和当前指令。
2. LLM 输出一个阶段常数出库流量。
3. 系统把这个出库流量保持到下一次触发时刻。
4. 下一阶段再基于更新后的状态继续调度。

所以，`S05` 当前实现并不是每天都重新规划，而是 4 个大阶段之间滚动更新。

## S05 的时间轴

### T0

- 触发时刻：`0 h`
- LLM 看到的状态：初始状态 `160.0 m`
- 指令文本：在 15 天内把水位从 `160.0 m` 逐步降到 `156.5 m`
- 推进窗口：`72 h`，即 `3 天`
- 达标判断：当前阶段 `llm_outflow` 是否比上一状态记录的 `outflow` 更小

这里的自动评估条件是 `direction=decrease`，它比较的是：

- 当前阶段 `llm_outflow`
- 上一状态记录的 `outflow`

因为初始状态里的 `outflow` 默认等于 `initial_inflow=200`，所以 `T0` 想通过，实际上要求的是：

- `llm_outflow < 200`

这和文字语义“为了降水位应加大出库”并不完全一致，说明当前 `T0` 的方向型配置更像一个代码层面的占位判定，而不是最严格的物理语义判定。

### T1

- 触发时刻：`72 h`
- LLM 看到的状态：`T0` 推进 3 天后的状态
- 指令文本：台风提前，要求更快降到 `156.5 m`
- 推进窗口：`48 h`，即 `2 天`
- 达标判断：推进后水位是否到达 `156.5 +/- 0.3 m`

这里有一个很关键的实现细节。

虽然文本语义说的是“5 天内完成降水位”，但当前代码推进窗口不是按这句话里的 5 天算，而是按相邻触发时刻的差值算。

因为下一次触发是 `120 h`，所以 `T1` 实际只会推进：

- `120 - 72 = 48 h`

也就是 2 天。

### T2

- 触发时刻：`120 h`
- LLM 看到的状态：`T1` 结束后的状态
- 指令文本：台风偏移，恢复原计划节奏，继续平稳下泄
- 推进窗口：`120 h`，即 `5 天`
- 达标判断：当前阶段 `llm_outflow` 是否比上一阶段更小

因此，`T2` 当前自动判定的重点是“是否比上一阶段更收敛”，而不是“是否匹配某条目标水位曲线”。

### T3

- 触发时刻：`240 h`
- LLM 看到的状态：`T2` 推进后的状态
- 指令文本：最终目标改为降到 `155.5 m`
- 推进窗口：剩余 `120 h`，即 `5 天`
- 达标判断：推进后水位是否到达 `155.5 +/- 0.3 m`

因此，`S05` 的最后一个阶段会把目标从原来的 `156.5 m` 再次下调，形成一次明确的末段目标切换。

## 这个场景为什么说是“动态过渡”

`S05` 的核心不是单次指令，而是目标在同一个 15 天窗口内不断变化：

1. 一开始是按常规过渡期节奏降到 `156.5 m`
2. 中途因为台风提前，要求加速完成
3. 随后又因为台风偏移，恢复较平稳的节奏
4. 最后又把末目标进一步下调到 `155.5 m`

所以它测的是：在连续变化的风险判断下，LLM 是否能不断修正策略。

## 这个场景里状态如何向后传递

每个阶段结束后，系统都调用 `_advance_state()` 推进状态：

1. 按 `3 h` 时步推进。
2. 入流按场景里的常数 `200 m3/s` 处理。
3. 出流按本阶段 LLM 给出的常数值处理。
4. 得到新的末水位和末库容。

随后这些新状态会被直接注入下一阶段场景。

所以 `S05` 的每次指令更新都建立在前一阶段已经执行过的结果之上，而不是重新回到 `160.0 m` 起点。

## 当前实现里最值得注意的地方

`S05` 在文字层面最接近“滚动调度”，但当前代码仍然不是逐日滚动。

准确地说，它现在是：

- 4 个阶段
- 每阶段一个常数出库流量
- 阶段之间更新状态和目标

因此，它已经具备“动态阶段切换”的特征，但还不是严格意义上的“每天重规划”的滚动调度器。

另一个需要注意的点是：

- `T0` 和 `T2` 的 `direction=decrease` 在当前实现里检查的是“出库是否比上一阶段更小”

这和“为了更快降水位通常要加大出库”的自然语义不完全一致。因此，`S05` 是当前配置里最适合继续细化 `pass_condition` 的场景之一。
