# S03 动态执行流程说明

本文档描述的是当前代码里的实际执行方式，对应实现文件主要是 `experiments/dynamic_experiment.py`、`experiments/evaluation_metrics.py` 和 `experiments/paper_experiment_runner.py`。

## 场景定位

`S03` 表示极端洪水应急场景。

当前动态实验里，这个场景已经被改成更贴近应急过程的起调状态：

- 当前水位 `156.5 m`
- 初始库容 `32.51 亿m3`
- 当前入流 `1000 m3/s`
- 预报入流 `12000 m3/s`
- 时步 `1 h`
- 总时长 `72 h`

这意味着 `S03` 比其他场景更强调快速演变和频繁触发。

## 指令更新方式

`S03` 也是标准的阶段式执行：

1. 每个阶段先把上一阶段末状态注入场景。
2. 然后只给 LLM 当前这一条应急指令。
3. LLM 输出一个阶段性的出库流量。
4. 系统用这个常数出库流量推进到下一触发点。
5. 下一阶段再基于新状态继续决策。

所以，`S03` 的核心不是一次算完，而是连续应急响应。

## S03 的时间轴

### T0

- 触发时刻：`0 h`
- LLM 看到的状态：初始应急起调状态
- 指令文本：按分级规则执行应急调度，优先保障大坝安全
- 推进窗口：`6 h`
- 达标判断：本阶段 `llm_outflow` 是否比上一状态的 `outflow` 更大

这里上一状态的 `outflow` 默认等于初始 `initial_inflow`，也就是 `1000 m3/s`。

因此 `T0` 的实际通过条件很直接：

- `llm_outflow > 1000`

只要 LLM 在极端洪水开始时选择加大泄量，就会被判为方向正确。

### T1

- 触发时刻：`6 h`
- LLM 看到的状态：`T0` 推进 6 小时后的状态
- 指令文本：大坝监测异常，最大出库立即限制到 `8000 m3/s`，并说明约束冲突
- 推进窗口：`6 h`
- 达标判断：`best_effort`

`best_effort` 是 `S03` 最特殊的部分。

当前代码对它的判定逻辑是：

1. 主安全约束是推进后的末水位不能超过 `169.15 m`。
2. 主要目标是尽量满足 `llm_outflow <= 8000`。
3. 只要安全约束满足，`pass` 就会是 `True`。
4. 是否真正贴近 `8000`，通过 `partial_credit` 体现。

也就是说，`T1` 当前实现里是一个“安全优先、接近度次之”的困难任务。

这里还有一个重要细节：

虽然指令文本要求“说明约束冲突”，但当前自动化评估并不会检查 `final_decision_text` 里是否真的解释了冲突。现在记录的是：

- `pass`
- `partial_credit`
- `detail`

而不是对回答文本做语义审查。

### T2

- 触发时刻：`12 h`
- LLM 看到的状态：`T1` 结束后的状态
- 指令文本：监测恢复，解除 `8000 m3/s` 限制，全力泄洪
- 推进窗口：`24 h`
- 达标判断：本阶段 `llm_outflow` 是否比 `T1` 更大

因此，这一步检查的是“是否重新加大泄量”，而不是某个固定目标流量。

### T3

- 触发时刻：`36 h`
- LLM 看到的状态：`T2` 结束后的状态
- 指令文本：进入退水阶段，控制出库不超过 `6000 m3/s`
- 推进窗口：剩余 `36 h`
- 达标判断：`llm_outflow <= 6000`

这一步表示场景从“全力应急泄洪”切换到“恢复性退水调度”。

## 这个场景里为什么说是“动态应急”

`S03` 的阶段变化非常明显：

1. 先加大泄量应对洪水。
2. 中途因为坝体监测异常突然收紧到 `8000`。
3. 再在监测恢复后重新放开。
4. 最后进入退水收敛阶段。

因此，系统考察的不是单次调度能力，而是 LLM 对互相冲突的应急目标能否连续切换。

## 状态如何在阶段间传递

这个场景的时步是 `1 h`，所以 `_advance_state()` 会按 1 小时粒度推进。

每一阶段末态都会变成下一阶段的起始状态，包括：

- 水位
- 库容
- 当前入流

这使得 `S03` 里的每次调度都带有明显的历史依赖：

- `T1` 的困难程度取决于 `T0` 已经把水位推到什么位置
- `T2` 能否加大泄量，取决于 `T1` 期间状态被压到了哪里
- `T3` 的退水控制，则依赖 `T2` 是否已经把洪峰顶过去

## 当前实现里最重要的困难任务语义

`S03 T1` 不是普通的“超过上限就失败”。

当前代码明确允许这种情况：

- 如果为了坝安全，LLM 没有完全压到 `8000`，但仍把末水位控制在 `169.15 m` 以下
- 那么本阶段仍然会判定为 `pass=True`

与此同时：

- 越接近 `8000`，`partial_credit` 越高
- 越偏离 `8000`，`partial_credit` 越低

这就是 `S03` 当前版本里“困难任务宽松通过”的实际落地方式。
