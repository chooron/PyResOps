# 实验设计（与当前实现对齐）

## 1. 实验目标

本文拟回答以下三个研究问题：

- **RQ1：静态全信息条件下，LLM + MCP 工具链能否生成高质量且合规的水库调度方案？**
- **RQ2：当上级指令沿时间轴分阶段变化时，LLM + MCP 是否能够基于真实演化状态持续调整方案并完成新指令？**
- **RQ3：当预测与实际来水存在结构化偏差、运行状态持续演化时，静态执行、规则修正与 LLM 工具修正三类控制方式谁更能维持安全、有效、稳定的滚动调度？**

围绕这三个问题，论文实验不再按“脚本文件”组织，而按“能力层级”组织为三组主实验和一组补充实验。

## 2. 统一实验设置

### 2.1 被测系统

被测方法为本文提出的 **LLM + MCP + pyresops** 调度框架。Agent 通过工具调用而非直接臆造数字完成决策，核心工具包括：

- `get_reservoir_status`：读取当前水库状态
- `query_dispatch_rules`：查询调度规程
- `check_safety_constraints`：检查安全与约束
- `simulate_dispatch_program`：调用 pyresops 执行水量平衡仿真
- `evaluate_dispatch_result`：调用 pyresops 进行多目标评分
- `optimize_release_plan`：生成优化下泄方案

### 2.2 统一物理环境

- 研究对象：滩坑水电站
- 统一仿真内核：`pyresops SimulationEngine`
- 统一评估内核：`pyresops EvaluationService`
- 统一约束：死水位、正常蓄水位、最小生态流量
- 统一输出：出库流量、末水位、5 维评分、约束违反数、工具调用次数、决策时间

### 2.3 场景集合

实验覆盖 5 个典型调度场景：

| 场景  | 类型       | 代表任务          |
| --- | -------- | ------------- |
| S01 | 台汛期预泄    | 超汛限时预泄腾库      |
| S02 | 梅汛期错峰    | 控制下游断面流量      |
| S03 | 极端洪水应急   | 超标准洪水下保大坝安全   |
| S04 | 枯水期发电优化  | 满足生态流量前提下优化发电 |
| S05 | 梅台过渡期降水位 | 在时限内将水位降至汛限   |

## 3. 论文主实验设计

### 3.1 实验 A：静态全信息调度能力

#### 实验目的

验证在“已知完整状态 + 预报固定 + 无外部扰动”的理想条件下，LLM + MCP 是否能给出高质量的一次性调度方案。这一实验作为后续动态实验和滚动控制实验的能力下界与参照基线。

#### 实验流程

1. 输入场景初始状态、固定入流预报和目标约束。
2. Agent 按“状态查询 -> 规则查询 -> 安全检查 -> 仿真 -> 评估”的流程完成一次性分析。
3. 从 Agent 最终方案中提取目标出库流量。
4. 使用 pyresops 对该方案做统一仿真与评分。
5. 与人工规则基线进行对比。

#### 对比方法

- **Human Baseline**：基于调度规程的人工规则代理
- **LLM + MCP**：本文方法

#### 评价指标

- 综合评分 `overall_score`
- 五维分项评分：防洪、供水、发电、生态、合规
- 约束违反数
- 工具调用次数
- 决策时间
- 流程完整率

### 3.2 实验 B：指令驱动的多阶段动态适应能力

#### 实验目的

验证当上级指令不是一次给完，而是在时间轴上分阶段下达时，Agent 是否能够基于上一阶段仿真后的真实状态继续调整策略，而不是在静态假设下重复回答。

#### 核心机制

- 每个场景设置 4 个触发点：`T0/T1/T2/T3`
- 相邻触发之间由 pyresops 推进水库状态
- 下一阶段输入的初始状态等于上一阶段方案执行后的真实末态
- 评估目标不再是“总分越来越高”，而是“每次新指令是否达标”

#### 实验流程

1. 在 `T0` 时刻输入初始状态和首条上级指令。
2. Agent 生成阶段方案并输出出库流量。
3. 使用 pyresops 将该方案推进至下一触发时刻。
4. 在新的真实状态上注入下一条上级指令。
5. 重复上述过程直到 `T3`。

#### 特别场景

S03 的 `T1` 为“困难任务”：入库洪峰快速抬升，同时上级要求将最大出库限制为 8000 m3/s。该任务物理上可能与大坝安全发生冲突，因此不能以简单的二元失败来评价。

#### 评价指标

- 阶段指令达标率
- 响应方向正确率
- 阶段执行期约束违反数
- 困难任务部分得分 `partial_credit`
- 全场景平均通过率

阶段通过率定义为：

```text
PassRate_s = (1 / K) * Σ I(stage_k pass)
```

其中 `K` 为该场景触发阶段数。

### 3.3 实验 C：预测-实际偏差场景下的滚动控制能力对比

#### 实验目的

实验 C 不再测试“系统是否会随着预报更新自动切换方案”，而是测试：当预测与实际来水存在结构化偏差、且运行状态持续演化时，静态执行、规则型修正和 LLM 调用现有调度工具三种控制方式，谁更能在统一规程与安全约束下维持安全、有效、稳定的滚动调度。

#### 适用场景

- S02：洪水错峰调度，重点考察洪峰时刻偏差和峰值量级偏差下的防洪-生态平衡
- S04：枯水期发电优化，重点考察逐旬偏丰/偏枯和相位错配下的发电-生态平衡

#### 对比控制方式

- `static` / Static Plan：初始时刻生成一份方案，后续固定执行，不做修正
- `rule_based` / Rule-based Adjustment：按既有调度规程和阈值规则做确定性修正
- `llm_tool` / LLM Tool-based Control：在相同规则边界内，由 LLM 通过 MCP/pyresops 工具完成滚动修正

#### 偏差场景设计

- 不再使用随机预报扰动，而是使用可复现的结构化偏差场景
- S02 设置 8 个偏差子场景：`D0` 无偏差基线、洪峰提前/滞后、峰值增大/减小，以及时间-量级联合偏差
- S04 设置 6 个偏差子场景：`D0` 无偏差基线、单旬偏枯、连续偏枯，以及“前丰后枯 / 前枯后丰”相位偏差
- 每个偏差场景都同时给定 `forecast_sequence` 与 `actual_inflow_sequence`，用于区分“控制器看到的预测”和“驱动真实状态推进的实际来水”

#### 运行流程

1. 在 `T0` 基于预测来水生成初始方案。
2. 用实际来水推进真实水库状态，而不是沿用预测状态。
3. 在固定周期触发或安全/偏差触发下，允许控制器继续沿用、规则修正或工具修正当前方案。
4. 对每一步执行结果进行统一仿真与评估，并记录修正是否有效。
5. 在全部偏差场景上汇总三类控制方式的均值表现与最差情形。

#### 评价指标

- 安全性：`total_constraint_violations`、`max_level_exceedance`、`has_critical_risk`
- 效果性：关键维度得分、`overall_score`
- 鲁棒性：相对无偏差基线 `D0` 的性能下降幅度 `performance_degradation`
- 修正能力：`correction_count`、`effective_correction_rate`
- 恢复能力：`recovery_steps`

| Human Baseline         | 实验 A 中对比传统人工规则调度能力              | 已实现    |

另：当前仓库中的 `ablation_study.py` 与 `statistical_analysis.py` 更偏论文补充材料或占位脚本，其中部分数据仍为模拟数据，不应在主实验中直接宣称为真实结果。

## 5. 论文版结果报告协议

为了让实验节更规范，建议论文正式结果按如下协议汇报：

1. 所有方法共享同一套 pyresops 物理仿真和评分器。
2. 静态实验报告 5 个场景的逐场景结果与均值。
3. 动态实验报告逐阶段达标率，而不是简单累加综合得分。
4. 滚动控制实验按控制器类型报告关键维度得分、`overall_score`、约束违反数、有效修正率和恢复步数，并同时给出均值与最差情形。
5. S02 与 S04 的每个偏差子场景分别报告；`D0` 作为无偏差基线，其余子场景额外报告 `performance_degradation`。
6. 若涉及 LLM 控制器的重复运行，应固定 `llm_seed` 与温度并报告均值与标准差；偏差序列本身采用确定性配置，不再依赖随机预报扰动。

若论文需要统计显著性分析，建议：

- 对静态实验使用 Wilcoxon 符号秩检验比较 LLM + MCP 与人工基线
- 对滚动控制实验可在偏差场景集合上报告 bootstrap 95% 置信区间，并补充 mean/worst-case 结果
- 对多模型配置结果补充效应量，而不只报告 p 值

## 6. 可直接写入论文的方法描述

可在论文实验节中使用如下表述：

> 为系统评估所提出的 LLM + MCP 水库调度框架，本文从静态决策、指令驱动动态适应和预测-实际偏差场景下的滚动控制三个层次构建实验体系。静态实验用于衡量模型在完整信息条件下的一次性调度能力；动态实验通过沿时间轴分阶段下达上级指令，评估模型在真实状态演化下的连续响应能力；滚动控制实验则在预测与实际来水不一致的结构化偏差场景下，比较静态执行、规则修正与 LLM 工具修正三类控制方式的安全性、有效性与鲁棒性。所有实验均共享同一套 pyresops 物理仿真内核与评估器，以避免由于评价口径不一致导致的偏差。

## 7. 当前代码撰写思路总结

### 7.1 总体思路

当前代码不是把“论文实验”写成一段大脚本，而是拆成四层：

1. **场景层**：场景参数统一用字典快照表示，便于在动态和滚动控制实验中做运行时注入。
2. **工具层**：`paper_experiment_runner.py` 将 pyresops 能力包装成 Agent 可调用的 MCP 风格工具。
3. **控制层**：`dynamic_experiment.py` 与 `automated_experiment.py` 分别实现多阶段动态控制与偏差场景滚动控制逻辑。
4. **评估层**：`evaluation_metrics.py` 统一负责真实仿真后的评分和达标判定。

这意味着代码的核心目标不是“让 LLM 直接给答案”，而是“让 LLM 在统一仿真-评估回路中做决策”。

### 7.2 逐文件理解

#### `experiments/paper_experiment_runner.py`

- 维护统一场景集合 `SCENARIOS`
- 提供 6 个工具的封装
- 负责模型配置加载和 Agent 初始化
- 统一收集 `outflow`、工具调用次数、最终文本和总耗时

这一层的核心意图是：**把模型接入问题、工具接入问题和实验入口问题放在一起解决**。

#### `experiments/evaluation_metrics.py`

- 统一定义滩坑水电站规格 `_build_tankan_spec`
- 统一用 `_run_pyresops_eval` 做仿真与评分
- 为动态实验额外提供 `evaluate_instruction_compliance`

这一层的核心意图是：**所有方法共用同一把尺子**。

#### `experiments/dynamic_experiment.py`

- 用 `DYNAMIC_TRIGGERS` 表达分阶段指令序列
- 用 `_advance_state` 在阶段间推进真实状态
- 用 `_apply_trigger` 把当前阶段的状态与指令重新注入场景快照
- 用 `run_multi_round_dynamic_experiment` 执行完整阶段循环

这一层的核心意图是：**把“时间推进”显式建模，而不是把多轮问答当成动态实验**。

#### `experiments/automated_experiment.py`

- 实验 C 仅保留偏差场景控制器与 `run_deviation_experiment()` / `run_all_deviation_experiments()` 主链入口
- 用 `DeviationScenarioSimulator` 生成结构化偏差场景下的 `forecast_inflow` / `actual_inflow` 序列
- 用 `StaticPlanController`、`RuleBasedAdjustmentController`、`LLMToolBasedController` 实现三类滚动控制器
- 用 `run_deviation_experiment()` 与 `aggregate_results()` 统一执行和汇总偏差场景结果

这一层的核心意图是：**把实验 C 写成“结构化偏差场景下的控制器对比”，而不是“随机扰动下的单一自动切换”**。

#### `experiments/baseline_human.py`

- 提供人工规则基线
- 虽然决策逻辑是规则化近似，但仿真和评分仍走 pyresops

这一层的核心意图是：**保证对比时，差异来自决策方式，而不是来自不同的物理求解器**。

### 7.3 当前代码的几个关键设计判断

当前实现里有几个很明确的设计取向：

- **先统一评估，再扩展实验**：所以评分逻辑被集中在 `evaluation_metrics.py`
- **先保证真实状态演化，再谈动态能力**：所以动态实验先做状态推进，再做下一阶段决策
- **让实验 C 从“是否切换方案”转向“在偏差场景下比较三类控制器的鲁棒性”**：重点不再是自动切换本身，而是安全、效果与修正能力的对比
- **尽量兼容旧接口**：因此实验 C 在引入 `DeviationScenarioSimulator` 和 `RollingControlResult` 的同时，仍保留旧 automated 入口

### 7.4 从代码角度看，当前实现最适合怎么写论文

最适合的写法不是强调“模型一次答得多聪明”，而是强调三点：

1. **工具增强决策**：关键数字来自工具和仿真，不是语言模型臆测。
2. **闭环状态更新**：动态与滚动控制实验都基于真实状态滚动推进。
3. **任务相关与鲁棒性评价**：不同实验用不同指标，实验 C 额外强调性能下降、修正有效性与恢复能力。

### 7.5 后续若继续打磨，最值得补的工程点

- 将场景与触发配置进一步外置到 YAML/JSON，降低脚本分叉
- 让 Agent 输出结构化 JSON，替代当前基于正则的 `outflow` 抽取
- 增加重复试验驱动器，直接输出均值、标准差和置信区间
- 统一静态、动态与滚动控制实验的结果 schema

## 8. 一句话结论

如果把当前代码转化为论文叙事，最合理的主线是：**本文不是只验证 LLM 会不会做一次调度，而是验证它是否能在统一仿真约束下完成静态决策、动态响应和偏差场景下的滚动控制三类能力。**
